# Market Research

## 核心原则

把调研作为决策支持，而不是目录填空。

按固定顺序执行：

```text
识别模式 → 校准决策 → 侦察证据 → 选择结构 → 深挖证据
→ 组织判断 → 核验 → 生成 MD/HTML → 验证交付
```

证据不足时收窄结论、转为盲区或验证计划。不要通过增加篇幅掩盖信息缺口。

## 选择运行模式

| 模式 | 触发信号 | 交付 |
|---|---|---|
| 快速判断 | “简单说说”“你怎么看”“先给个判断” | 对话内简答；注明依据和限制，不创建文件 |
| 标准调研 | 市场机会、行业分析、竞品、进入策略、市场空间、招投标格局 | 自适应 Markdown、HTML、evidence YAML |
| 行业入门 | 用户明确表示不懂、需要从零建立认知 | 基础认知 + 决策分析的 Markdown、HTML、evidence YAML |

销售拜访准备由独立 `sales-visit-prep` Skill 处理。学术文献综述使用学术研究 Skill。

## 1. 校准决策问题

开始搜索前，确认以下最小信息：

1. 行业或产品范围。
2. 决策目的：进入、产品方向、竞争、投资、销售材料或其他。
3. 地域、客户类型和时间范围；仅在它们会改变结论时追问。

将“有没有机会”“空间大不大”等模糊请求重写为可验证问题，并说明回答范围。

优先使用交互式澄清工具。无交互工具、API、Cron 或自动评测环境中，用一条文本集中提出必要问题并停止；不要等待不可用的交互回调。用户已经给足信息时直接执行，不重复确认。

## 2. 侦察证据

在拟定报告目录前执行短侦察，至少覆盖三个可能改变决策的证据通道。记录查询式、时间窗、可用结果、来源类型、质量和是否继续。

输出证据地图：

| 方向 | 查询/来源 | 密度 | 质量 | 决策影响 | 动作 |
|---|---|---|---|---|---|
| 示例 | 政策、采购、财报 | 高/中/低 | 高/中/低 | 高/中/低 | 深挖/谨慎/盲区/放弃 |

不要先写固定目录。详细侦察方法和网页安全规则见 `references/evidence-standard.md` 与 `references/source-policy.md`。

## 3. 选择研究结构

根据决策问题和证据地图选择章节，不根据 ToG/ToB/ToC 套固定模板。

- 标准调研：读取 `references/standard-research.md`。
- 行业入门：读取 `references/industry-onboarding.md`。
- 报告成文与引用：读取 `references/report-writing.md`。

只保留能回答决策问题或构成关键盲区的章节。标准报告通常包含 4-7 个主体章节；这不是硬性字数或目录要求。

## 4. 建立结构化证据

为正式报告维护 `{topic}.evidence.yaml`。每条来源、判断和假设都使用稳定 ID：

- 来源：`S1`、`S2`……
- 判断：`C1`、`C2`……
- 正文引用：`[S1]`

区分 `fact`、`inference`、`judgment`、`recommendation`。推断和判断不得写成事实。

运行：

```bash
python scripts/validate_evidence.py <evidence.yaml>
```

证据 schema、同源去重和置信度规则见 `references/evidence-standard.md`。

## 5. 深挖高价值通道

只深挖证据地图中“决策影响高且可验证”的方向。

最低要求：

- 重大数字、日期、公司名、政策名和比较级判断回到原始来源核验。
- 高置信度判断至少有两个独立来源，且至少一个是原始来源。
- 厂商页面只能证明厂商主张，不能单独证明市场事实。
- 转载和同一新闻稿不算独立来源。
- 公开资料无法回答的变量，说明代理信号和验证路径。

## 6. 撰写与生成交付物

正式报告默认生成：

```text
<topic>-调研报告.md
<topic>-调研报告.html
<topic>-evidence.yaml
```

输出目录优先级：用户指定目录 → 当前工作区 `outputs/` → 系统 Downloads。

HTML 必须由成熟 Markdown 解析器和唯一模板生成：

```bash
python scripts/render_report.py <report.md> <report.html> --theme light
```

`light` 是适合长文、打印和客户转发的默认主题；`dark` 仅在用户要求时使用。不要手写第二套 HTML 或复制旧模板。

## 7. 核验并交付

交付前运行：

```bash
python scripts/validate_report.py <report.md> --evidence <evidence.yaml> --html <report.html>
```

只有同时满足以下条件才算完成：

- 研究问题、范围和截止日期明确。
- 重大判断能追溯到结构化来源。
- 数字、日期、公司、政策和比较级表述已核验。
- 事实、推断、判断和建议没有混写。
- 盲区包含具体验证路径。
- Markdown 与 HTML 内容一致，引用和外链可点击。
- HTML 语法、目录、表格和移动端布局通过验证。

对话中给出核心判断、关键限制和可点击文件路径，不粘贴整份报告。

## 常见失败

- 用户已经给足上下文，仍重复追问。
- 把“提出调研”默认理解为“完全不懂行业”。
- 先决定目录，再寻找能填满目录的材料。
- 用固定篇幅、固定样本数代替证据质量。
- 把多个转载当成多个独立来源。
- 把内部证据卡直接当作交付物，而不是组织成连贯报告。
- 只生成 Markdown，遗漏 HTML 或 evidence YAML。
- 在最终报告中保留开发对话、纠错经过、与研究无关的个人信息或 AI 自我提醒。
