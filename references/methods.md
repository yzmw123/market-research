# Methods

Use methods as analysis lenses, not mandatory report sections. Apply only when evidence supports the method.

## Method Selection

| Situation | Use this lens | Avoid |
|---|---|---|
| Need a stronger formal report architecture | Report directory mining + adaptive module selection | Copying a paid report TOC as the final outline |
| Digital economy / AI / data governance topic | CAICT-style institutional synthesis | Treating consulting/blog material as equal authority |
| User asks whether a direction is worth doing | DBS problem dissolution + market opportunity judgment | Generic “industry overview” |
| Evidence is rich but scattered | Signal synthesis | Fixed PESTLE table |
| Product pages/docs/pricing are rich | Product/competitor teardown | Feature checklist only |
| User feedback is rich | Sentiment + JTBD | Treating vocal users as all users |
| Market size data is credible | TAM/SAM/SOM range | Single precise market-size number |
| Public sources are weak | Hypothesis + validation plan | Fake full report |
| Product idea is new | XYZ hypothesis + Pretotype | Asking “would you use this?” |
| Plan needs challenge | gstack-style premise challenge | Accepting the user's framing blindly |

## Report Directory Mining Lens

Use when the report structure feels generic or the user points to professional report sites.

Steps:

1. Mine 2-5 relevant paid report TOCs (e.g. Cir.cn) for recurring modules.
2. Mine 3-5 short analysis articles (e.g. chinabgao.com/info) for table types and narrative progression.
3. Separate **structure evidence** from **fact evidence**.
4. Pick only modules that match the user's decision and available evidence.
5. Convert unsupported-but-important modules into blind spots or validation tasks.

Useful output: a Structure Scout table and a sharper adaptive outline.

## CAICT-style Institutional Synthesis Lens

Use when authoritative institutional reports are available, especially CAICT / 中国信通院 for digital and AI topics.

Steps:

1. Extract report preface and TOC first; CAICT prefaces often state the report logic.
2. Identify whether the report is governance, AI赋能产业, technology evolution, government digitalization, or enterprise application.
3. Reuse its structure pattern only if it matches the user's decision.
4. Cite CAICT for definitions, frameworks, maturity judgments, and cases; verify critical numbers elsewhere.
5. Translate institutional recommendations into actionable implications for the user.

Useful output: an authority-backed framework plus a grounded recommendation path.

## DBS Lens: Dissolve Bad Questions

Use when the request contains vague business words.

Ask:

- What would count as evidence that this is “worth doing”?
- Is the user asking a Question with a textual answer, or a Problem that needs practice/validation?
- What assumption would make the whole question disappear if false?

Useful output: a sharper research question and a list of what the research will not answer.

## Superpowers Lens: Gates

Use gates to prevent premature work:

1. No calibrated question → no research.
2. No evidence map → no report outline.
3. No sources → no factual claim.
4. No verification → no final delivery.

## gstack Lens: Premise Challenge

Before deep research, challenge the plan:

- Is this the right problem to research?
- Is there a more useful framing?
- Would a narrower segment produce stronger evidence?
- Are we trying to answer a private-information question with public data?
- Which part of the plan is template-driven rather than evidence-driven?

## Lenny / PM Lens: Competition Includes the Status Quo

For product and competitor research, include alternatives beyond direct competitors:

- Current manual process.
- Excel/spreadsheet/workaround.
- Legacy system.
- “No decision”.
- Outsourcing/service provider.
- Adjacent product category.

Ask: “If this product did not exist, what would the customer do?”

## PM-skills Lens: Market Sizing

Use TAM/SAM/SOM only when credible data exists.

Rules:

- Give ranges, not fake precision.
- Show top-down and bottom-up logic when possible.
- Label every assumption.
- If data is recycled across low-quality reports, lower confidence.
- If sizing does not affect the user's decision, do not over-invest in it.

## PM-skills Lens: Sentiment + JTBD

Use when enough user feedback exists.

Do:

- Cluster by job/scenario, not just positive/negative emotion.
- Extract direct quotes as evidence.
- Distinguish complaint frequency from complaint severity.
- Flag sample bias: public comments represent vocal users, not all users.

Do not infer retention, CAC, LTV, or silent churn from comments.

## PM-skills Lens: Opportunity Solution Tree

Use for product opportunity research when user wants to decide what to build.

Structure:

1. Desired outcome.
2. Opportunities / pains / unmet jobs.
3. Possible solutions.
4. Experiments to test assumptions.

Use public research to generate opportunities; use validation plan to test them.

## PM-skills Lens: XYZ Hypothesis + Pretotype

Use when public evidence is too thin or the idea is new.

Format:

> At least X% of Y will do Z.

Prefer behavior tests:

- Landing page signup.
- Paid preorder.
- Manual concierge MVP.
- Demo call booking.
- Waitlist with qualification.
- Fake-door click with clear follow-up.

Opinions are weak. Behavior with cost is stronger.
